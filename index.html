Chapter 1

Traditional (Lexical) Search

1. Definition

Lexical search (also called keyword or sparse search) is the classic "match the words" approach — the same idea behind Ctrl+F or a plain search bar. It looks for the exact words from your query inside the documents. To do that quickly, it builds an inverted index: picture the index at the back of a textbook that maps each word to the list of pages where it appears. It then sorts the matching documents with BM25, a scoring formula that (a) rewards documents that use your search words more often (term frequency), (b) gives extra weight to rare, specific words (inverse document frequency), and (c) stops very long documents from winning just because they contain more words (length normalization). Because it only matches the literal text and ignores meaning, it's the right tool when things must match exactly — product SKUs, error codes, function names, and IDs — where "close enough" is simply wrong.

Note

"Sparse" and "lexical" mean almost the same thing here. Sparse describes how the text is stored — as a long list of word counts that is mostly zeros, since most words never appear in any given document. Lexical describes the strategy — matching the exact words. BM25 is the standard formula for scoring those matches.

# lexical_search.py — tokenize, index, and rank a small corpus with BM25
import re
from dataclasses import dataclass

from nltk.corpus import stopwords
from nltk.stem import PorterStemmer
from rank_bm25 import BM25Okapi

_TOKEN_RE = re.compile(r"[A-Za-z0-9_]+")
_STOPWORDS = set(stopwords.words("english"))
_STEMMER = PorterStemmer()


def tokenize(text: str) -> list[str]:
    """Lowercase, split, drop stopwords, stem. Must be applied identically
    at index time and query time.
    e.g. tokenize("Connections were RESET.") -> ['connect', 'reset']"""
    tokens = _TOKEN_RE.findall(text.lower())
    return [_STEMMER.stem(t) for t in tokens if t not in _STOPWORDS]


DOCUMENTS = [
    {"id": "doc-1", "title": "Connection Reset Errors",
     "text": "The server returned ERR_CONN_RESET_499 when the client closed "
             "the connection before the response was fully sent."},
    {"id": "doc-2", "title": "Configuring Retry Policies",
     "text": "Retries should use exponential backoff with jitter. Configure "
             "max_retries and base_delay_ms in the client config."},
    {"id": "doc-3", "title": "Database Connection Pooling",
     "text": "Connection pooling reduces the overhead of establishing new "
             "database connections. Tune pool_size and max_overflow."},
    {"id": "doc-4", "title": "SKU Lookup API",
     "text": "The /v1/products/{sku} endpoint returns product metadata for "
             "a given SKU, such as SKU-88421-BLK."},
]


@dataclass
class SearchResult:
    doc_id: str
    title: str
    score: float


class LexicalSearchEngine:
    def __init__(self, documents: list[dict[str, str]], k1: float = 1.5, b: float = 0.75) -> None:
        self._documents = documents
        corpus_tokens = [tokenize(doc["text"]) for doc in documents]
        # BM25Okapi precomputes IDF and average doc length (avgdl) here.
        self._bm25 = BM25Okapi(corpus_tokens, k1=k1, b=b)

    def search(self, query: str, top_k: int = 5) -> list[SearchResult]:
        scores = self._bm25.get_scores(tokenize(query))
        ranked = sorted(range(len(scores)), key=lambda i: scores[i], reverse=True)[:top_k]
        return [
            SearchResult(self._documents[i]["id"], self._documents[i]["title"], round(float(scores[i]), 4))
            for i in ranked
            if scores[i] > 0
        ]


if __name__ == "__main__":
    engine = LexicalSearchEngine(DOCUMENTS)
    for r in engine.search("connection reset error", top_k=3):
        print(f"{r.score:6.3f}  {r.doc_id}  {r.title}")
    # -> 2.847  doc-1  Connection Reset Errors
    # -> 0.412  doc-3  Database Connection Pooling
  • Lower k1 (e.g. 1.0) — term frequency saturates fast; good for short, keyword-dense docs (logs, SKUs).
  • Higher k1 (e.g. 2.0) — repeated terms keep contributing; better for long-form documents.
  • Lower b (e.g. 0.3) — less length penalty, for corpora with legitimate length variance.
  • Higher b / b=1 — full length normalization, stops long docs winning purely on word count.
  • Field boosting — index title/body separately, combine per-field BM25 scores (BM25F), weighting title matches higher.
Warning

Aggressive stemming can hurt precision on exact-identifier queries. Keep a second, unstemmed field for SKUs/error codes/function names so exact matches aren't lost to stemming artifacts.

For a persistent inverted index at scale, use Whoosh (embedded) or OpenSearch/Elasticsearch (BM25 by default).

3. Production & Integration

Client → GET /search?q=...&top_k=5 → FastAPI → BM25 engine → JSON (ranked docs + scores)


# main.py — FastAPI service wrapping the lexical search engine
from fastapi import FastAPI, Query
from lexical_search import DOCUMENTS, LexicalSearchEngine

app = FastAPI(title="Lexical Search Service")
engine = LexicalSearchEngine(DOCUMENTS)  # built once at startup

@app.get("/search")
def search(q: str = Query(..., min_length=1), top_k: int = Query(5, ge=1, le=50)):
    hits = engine.search(q, top_k=top_k)
    return {"query": q, "results": [h.__dict__ for h in hits]}

In RAG, BM25 serves as the retrieval step: it fetches the top-k lexically relevant chunks that get concatenated into the LLM's context — a strong default when queries use exact terminology (API names, config keys, legal citations).

  • Storage/index — precompute at write time; inverted index lives on disk (Lucene segments) or in memory for small corpora.
  • Analyzers/tokenization — must be identical at index and query time (lowercasing, stopwords, stemming).
  • Sharding/scale — partition the index across nodes; replicate shards for fault tolerance and read throughput; a query fans out to all shards and results are merged.
  • Caching — filter/query caches at the shard level; request cache for identical repeated queries (e.g. dashboards).
  • Latency — track p50/p95/p99 query latency, indexing rate, and refresh interval (how soon writes become searchable, default ~1s).
Note

Precompute (index at write time) whenever the corpus changes slowly relative to query volume — the common case. Reserve query-time scoring (scripted fields) for attributes that change too fast to justify re-indexing; it bypasses the fast inverted-index path.

5. Summary

Lexical search matches query tokens against an inverted index and ranks candidates with BM25 — no embedding model or GPU, very low latency, fully explainable. It's the right default for exact identifier lookup, code search, and any workload where literal precision matters, but it cannot bridge synonyms or paraphrase — the gap dense vector search closes in the next chapter.

Chapter 2

Dense Vector Search

1. Definition

Dense vector search turns each document and each query into an embedding — a fixed-length list of numbers (a "vector") produced by a neural network. The easiest way to picture it: each vector is a point in space, and things with similar meaning sit close together. Finding relevant text then becomes a geometry problem — find the document points nearest to the query point, measured with cosine similarity or dot product. The big advantage is that it captures meaning, not just words, so "car" and "automobile" land near each other even though they share no letters. That's exactly what powers semantic search and Retrieval-Augmented Generation (RAG). Checking the query against every stored vector one by one is accurate but slow once you have millions or billions of them, so real systems use Approximate Nearest Neighbor (ANN) indexes (HNSW, IVF, PQ) that find the near-matches far faster, accepting a tiny drop in accuracy for a big speed gain.

2. Development

pip install sentence-transformers faiss-cpu numpy

Encode a small corpus with BAAI/bge-small-en-v1.5 (384-dim, fast on CPU), L2-normalize so inner product equals cosine similarity, build an exact IndexFlatIP baseline, and query it: 

"""dense_search.py — encode, index, and query a small corpus."""

from sentence_transformers import SentenceTransformer
import faiss

MODEL_NAME = "BAAI/bge-small-en-v1.5"

CORPUS = [
    "FAISS is a library for efficient similarity search of dense vectors.",
    "HNSW builds a multi-layer graph for approximate nearest neighbor search.",
    "Qdrant is an open source vector database written in Rust.",
    "Cosine similarity measures the angle between two vectors, ignoring magnitude.",
    "Retrieval-Augmented Generation grounds LLM answers in retrieved documents.",
    "Product quantization compresses vectors to reduce memory footprint.",
]

model = SentenceTransformer(MODEL_NAME)

# normalize_embeddings=True does L2 normalization inline — required so
# IndexFlatIP (inner product) ranks by pure cosine similarity.
vectors = model.encode(
    CORPUS, normalize_embeddings=True, convert_to_numpy=True
).astype("float32")

index = faiss.IndexFlatIP(vectors.shape[1])   # exact, brute-force cosine search
index.add(vectors)


def search(query: str, k: int = 3) -> list[tuple[str, float]]:
    query_vector = model.encode(
        [query], normalize_embeddings=True, convert_to_numpy=True
    ).astype("float32")
    scores, ids = index.search(query_vector, k)
    return [(CORPUS[i], float(s)) for s, i in zip(scores[0], ids[0]) if i != -1]


for doc, score in search("database for storing embeddings"):
    print(f"[{score:.4f}] {doc}")
Note

For millions of vectors use IndexHNSWFlat / IndexIVFFlat, or a vector DB. ChromaDB (simplest local) and Qdrant (production, metadata filtering) offer the same flow with persistence.

3. Production & Integration

Retrieval is typically exposed as an HTTP service and used as the retrieval stage of a RAG pipeline: 

"""app.py — expose dense retrieval over HTTP."""

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()
model = SentenceTransformer(MODEL_NAME)
index = faiss.read_index("index.faiss")   # built offline by an ingestion job


class SearchRequest(BaseModel):
    query: str
    k: int = 5


@app.post("/search")
def search_endpoint(req: SearchRequest):
    qv = model.encode([req.query], normalize_embeddings=True).astype("float32")
    scores, ids = index.search(qv, req.k)
    return [{"id": int(i), "score": float(s)} for s, i in zip(scores[0], ids[0]) if i != -1]

RAG pipeline: retrieve top-k chunks -> assemble a grounded prompt -> call the LLM -> return a cited answer. 

Ingest: Documents -> Chunk -> Embed (batched, GPU) -> Vector DB (HNSW/IVF)
Query:  Client -> Embed Query (cached) -> ANN Search + Filter -> Top-K -> LLM -> Answer
  • Embedding service — batch texts per forward pass on GPU workers, decoupled from query traffic.
  • Chunking — 200–500 tokens with 10–20% overlap is a common starting point.
  • Vector DB choice — Qdrant, Milvus, or pgvector for replication, sharding, and payload filtering beyond a bare FAISS process.
  • HNSW/IVF params — M/ef_construction (or nlist) at build time; ef_search/nprobe tunable per-request for the recall/latency tradeoff.
  • Incremental upserts — update by id rather than full rebuilds; schedule periodic reindexing since ANN graphs degrade under heavy churn.
  • Caching — cache query embeddings and repeated search results to cut embedding and ANN load.

5. Summary

Dense vector search encodes text into embeddings and retrieves by cosine/dot-product similarity, using ANN indexes (HNSW, IVF, PQ) to stay fast at scale — matching on meaning rather than exact wording, which makes it the backbone of semantic search and RAG. Build it starting from a normalized IndexFlatIP baseline in FAISS, then scale through HNSW/IVF tuning to a managed vector database like Qdrant, served behind a retrieval API that a RAG pipeline calls on every query.

Chapter 3

Learned Sparse Retrieval

1. Definition

Learned sparse retrieval is a hybrid: it keeps the fast, word-based machinery of keyword search but lets a neural network decide how important each word is, instead of a hand-written formula like TF-IDF or BM25. The best-known model is SPLADE (SParse Lexical AnD Expansion). For every document and query it produces a sparse {term_id: weight} list — a weight for just a handful of words out of the full ~30,000-word vocabulary, and zero for all the rest. Its clever trick is term expansion: it can switch on related words that never actually appear in the text (for example, "heart attack" also lights up "cardiac" and "myocardial"), giving you some of the meaning-awareness of dense search. And because only a few hundred words have a nonzero weight, the output drops straight into the same proven keyword-search engines (Lucene, Elasticsearch, OpenSearch, Qdrant sparse vectors) — no special vector (ANN) index required.

Note

Think of learned sparse retrieval as the middle ground between two extremes: classic keyword search (BM25 — fast and exact, but blind to meaning) and dense vector search (smart about meaning, but a "black box" that needs a special vector index). That's why it's nicknamed "neural lexical search" — a neural network judges relevance, but the result stays a plain, human-readable list of weighted words.

2. Development

pip install transformers torch

Load AutoModelForMaskedLM (not plain AutoModel) because SPLADE needs the MLM head that projects token states into vocabulary-sized logits. Encoding pools those logits into a sparse vector via max_t log(1 + relu(logits[t])) over the sequence dimension, then an inverted index and dot-product scorer make it searchable. 

# splade.py
from collections import defaultdict

import torch
from transformers import AutoModelForMaskedLM, AutoTokenizer

MODEL_NAME = "naver/splade-v3"


class SpladeEncoder:
    def __init__(self, model_name: str = MODEL_NAME) -> None:
        self.device = "cuda" if torch.cuda.is_available() else "cpu"
        self.tokenizer = AutoTokenizer.from_pretrained(model_name)
        self.model = AutoModelForMaskedLM.from_pretrained(model_name).to(self.device).eval()

    def encode(self, text: str) -> dict[int, float]:
        """logits -> log(1 + relu(x)) -> max-pool over tokens -> sparse {term_id: weight}."""
        tokens = self.tokenizer(text, truncation=True, max_length=256, return_tensors="pt").to(self.device)
        with torch.no_grad():
            logits = self.model(**tokens).logits[0]          # [seq_len, vocab_size]
        activated = torch.log1p(torch.relu(logits))          # log(1 + relu(x))
        sparse_vec = torch.max(activated, dim=0).values       # max-pool -> [vocab_size]
        nonzero = torch.nonzero(sparse_vec).squeeze(-1)
        return {int(i): float(sparse_vec[i]) for i in nonzero.tolist()}

    def decode_terms(self, vec: dict[int, float], top_k: int = 10) -> list[tuple[str, float]]:
        ranked = sorted(vec.items(), key=lambda kv: kv[1], reverse=True)[:top_k]
        return [(self.tokenizer.decode([tid]).strip(), round(w, 3)) for tid, w in ranked]


class SparseInvertedIndex:
    def __init__(self) -> None:
        self.postings: dict[int, list[tuple[str, float]]] = defaultdict(list)
        self.doc_vectors: dict[str, dict[int, float]] = {}

    def build(self, encoder: SpladeEncoder, corpus: dict[str, str]) -> None:
        for doc_id, text in corpus.items():
            vec = encoder.encode(text)
            self.doc_vectors[doc_id] = vec
            for term_id, weight in vec.items():
                self.postings[term_id].append((doc_id, weight))


def search(query: str, encoder: SpladeEncoder, index: SparseInvertedIndex, top_k: int = 5):
    query_vec = encoder.encode(query)
    scores: dict[str, float] = defaultdict(float)
    for term_id, q_w in query_vec.items():                    # dot product, postings-scoped
        for doc_id, d_w in index.postings.get(term_id, []):
            scores[doc_id] += q_w * d_w
    return sorted(scores.items(), key=lambda kv: kv[1], reverse=True)[:top_k]


if __name__ == "__main__":
    encoder = SpladeEncoder()
    corpus = {
        "doc1": "Patient presented with symptoms of a heart attack.",
        "doc2": "Myocardial infarction requires immediate cardiac intervention.",
        "doc3": "The new laptop features a faster processor and longer battery life.",
    }
    index = SparseInvertedIndex()
    index.build(encoder, corpus)

    for doc_id, score in search("heart attack symptoms", encoder, index, top_k=3):
        print(f"{doc_id}  score={score:.3f}")
    # doc1 ranks highest on literal overlap; doc2 ranks above doc3 purely via
    # expansion ("cardiac" activates from "heart attack" context), which a
    # raw BM25 index matching only literal tokens would miss entirely.
Tip

Printing decode_terms() on sample documents is the fastest sanity check: if top terms look like noise (stopword or subword fragments), the checkpoint/tokenizer pairing or truncation is likely wrong. Distilled backbones (e.g. naver/splade-cocondenser-ensembledistil) cut query-side latency roughly in half; a FLOPS regularization term at training time controls the recall/sparsity tradeoff of a given checkpoint.

For production, store precomputed sparse vectors in Qdrant (native SparseVector), Elasticsearch rank_features, or OpenSearch neural sparse — all score with an exact dot product over matching postings, identical relevance to the in-memory index above.

3. Production & Integration

Sparse retrieval splits cleanly into an expensive offline stage and a cheap online stage: document vectors are precomputed once, in batches, on GPU; query vectors are encoded per-request on a low-latency path. Wrap the query encoder in a FastAPI service and fuse with BM25 and/or a dense retriever — SPLADE's expansion, BM25's exact-match precision, and dense paraphrase-level semantics rarely fail on the same queries. 

 OFFLINE (batch, GPU)                    ONLINE (per request)
 Corpus -> SPLADE doc encoder            Query -> SPLADE query encoder (distilled)
   -> sparse vectors -> bulk load into      -> postings lookup + dot-product score
      Qdrant / rank_features / neural-sparse -> RRF fuse w/ BM25 -> cached response

# service.py
from fastapi import FastAPI
from pydantic import BaseModel

from splade import SpladeEncoder, SparseInvertedIndex, search as sparse_search

app = FastAPI()
encoder = SpladeEncoder()
index = SparseInvertedIndex()  # index.build(encoder, load_corpus()) at startup


class SearchRequest(BaseModel):
    query: str
    top_k: int = 10


@app.post("/search")
def search_endpoint(req: SearchRequest):
    results = sparse_search(req.query, encoder, index, top_k=req.top_k)
    return [{"doc_id": d, "score": s} for d, s in results]

For hybrid search, fuse SPLADE's ranking with BM25 or a dense retriever via Reciprocal Rank Fusion (RRF) — it needs no score calibration across incomparable scales, only rank position.

  • Precompute document vectors offline on GPU; re-encode only on create/update, never per query.
  • Store postings in an existing inverted index (Qdrant sparse, Elasticsearch, OpenSearch) rather than adding a new system.
  • Encode queries synchronously at runtime with a distilled/smaller checkpoint and a capped sequence length to keep p99 low.
  • Cache encoded query vectors and/or result sets keyed on normalized query text — encoding is the dominant per-request cost.
  • Expansion multiplies term count per document versus raw tokenization; plan storage and reindex windows accordingly.

5. Summary

Learned sparse retrieval, exemplified by SPLADE, replaces BM25's fixed term-frequency heuristics with a neural model that expands documents/queries into semantically related vocabulary while keeping output sparse and inverted-index-compatible. The practical tradeoff is compute over lexical tokenization, so production systems precompute document vectors offline on GPU, encode queries on a low-latency path, store postings in existing search engines or sparse-native vector databases, and typically fuse results with BM25 and/or dense retrieval — the strongest fit for mature lexical infrastructure and interpretable first-stage retrieval, a poor fit for tiny corpora or extreme low-latency budgets.

Chapter 4

Multi-Vector Search

1. Definition

Most vector search squeezes a whole document down into a single embedding. Multi-vector search — made popular by ColBERT and extended by BGE-M3 — instead keeps one embedding per token (roughly, per word), so a document becomes a whole set of vectors rather than one averaged-out vector. It compares them using late interaction: the query and the document are each encoded on their own first (which is fast and can be cached), and only afterward are their token vectors matched up. The scoring rule is MaxSim: for each query word, find its best-matching word in the document, then add up those best matches — score(Q, D) = Σ_i max_j cos(q_i, d_j). Keeping every token means the fine details aren't blurred away by averaging, which is why it's the top choice for long, many-topic documents and the go-to second-pass reranker in high-precision RAG pipelines.

2. Development

MaxSim's entire mathematical core in PyTorch — two L2-normalized token matrices, a matmul, a row-wise max, and a sum:


import torch

def maxsim_score(query_emb: torch.Tensor, doc_emb: torch.Tensor) -> torch.Tensor:
    # query_emb: [n_q, dim], doc_emb: [n_d, dim] — both L2-normalized
    sim = query_emb @ doc_emb.T                     # cosine sim, [n_q, n_d]
    max_sim_per_query_token = sim.max(dim=1).values  # best doc token per query token
    return max_sim_per_query_token.sum()

For real use, ColBERT via RAGatouille wraps the ColBERTv2 checkpoint and its PLAID index behind a high-level API:

In production ColBERT is normally used as a reranker over a first-stage retriever's candidates via RAG.rerank(query=..., documents=candidate_docs, k=3) rather than searching the full corpus directly.

BGE-M3 (FlagEmbedding) returns dense + sparse + multi-vector in one pass; call model.colbert_score() on the returned colbert_vecs to get a late-interaction score without deploying a separate ColBERT model.

Note

PLAID + centroids. Storing every token vector uncompressed does not scale. PLAID clusters token embeddings across the corpus into centroids (k-means) and stores each token as a centroid id plus a quantized residual, pruning candidates by cheap centroid similarity before running exact MaxSim on the survivors.

3. Production & Integration

The canonical pattern: a cheap first-stage retriever (BM25 or single-vector dense) produces ~100 candidates, ColBERT MaxSim reranks them, and the top 5 go to the LLM — typically wrapped as a FastAPI endpoint:


@app.post("/search", response_model=list[SearchHit])
def search(req: SearchRequest) -> list[SearchHit]:
    candidates = bm25_search(req.query, k=req.candidate_pool)             # Stage 1
    reranked = rerank_candidates(req.query, candidates, top_k=req.top_k)  # Stage 2
    return [SearchHit(content=c["content"], score=c["score"]) for c in reranked]

              │

RAG note

Using ColBERT/BGE-M3 as the reranking stage before context reaches the LLM materially improves context selection: token-level matching surfaces the chunk that actually contains the answer, rather than the chunk whose average embedding happens to be closest to the query.

  • Storage cost. A 500-token document at 128 dims is ~256 KB uncompressed (500 × 128 × 4 bytes) vs. ~3 KB for a single pooled vector — roughly a 50-200x increase in stored vectors overall. PLAID-style centroid + residual compression typically claws back 10-30x, but multivector indexes still need far more RAM/SSD than single-vector ones at the same corpus size.
  • PLAID index. Centroid pruning + ANN narrows candidates before exact MaxSim runs, keeping the compressed index queryable at scale.
  • Two-stage retrieval. Only the reranking stage needs full-precision multi-vector data — the first-stage index stays single-vector and cheap, bounding latency as the corpus grows.
  • GPU/latency. Encoding and Stage-2 MaxSim want GPU batching; Stage-1 candidate search runs fine on CPU. Precompute and cache document token embeddings at index time — only the query is encoded on the request path.

5. Summary

Multi-vector search replaces one pooled embedding per document with one embedding per token, and replaces a single dot product with MaxSim late interaction, preserving detail that single-vector averaging destroys. The practical cost is storage and latency, addressed in production with PLAID-style compression, native multivector stores (Qdrant, Vespa), and — almost universally — a two-stage pipeline where a cheap first-stage retriever narrows the field and ColBERT or BGE-M3 late interaction reranks a small candidate set for the final answer.

Chapter 5

Graph-Based Retrieval

1. Definition

Graph-based retrieval stores your data as a property graph — nodes for things (people, companies, products) joined by labeled, directional edges that describe how they relate (WORKS_AT, PRODUCES), where both nodes and edges can carry extra key-value details. To answer a question, it follows the connections outward from a starting node instead of measuring how similar two pieces of text are. That makes it perfect for "who is connected to what" questions — "who else works where Alice used to work?", "is this transaction linked to a known fraud ring?" — which are answered by hopping across relationships. A vector index simply can't do this: it can tell you two documents are on similar topics, but not that two specific entities are exactly two hops apart. GraphRAG brings this to LLM apps: instead of dumping the top-k similar text chunks into the prompt, it pulls out a small, structured, explainable slice of the graph and hands that to the model as context.

2. Development

This section builds a graph retrieval system on Neo4j with the official neo4j Python driver.

Setup

docker run --name neo4j -p 7474:7474 -p 7687:7687 -e NEO4J_AUTH=neo4j/password -d neo4j:5

Install with pip install neo4j fastapi uvicorn anthropic. The driver pools connections — create one Driver per app and reuse it. Query parameters ($id, $name) can only stand in for values, never for labels, relationship types, or property keys — those are part of the query's structure and must be validated (e.g. against an allow-list) before string interpolation.

"""db.py — driver, parameterized helper, MERGE nodes + relationship, multi-hop MATCH."""
from neo4j import GraphDatabase

driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))


def run_query(query: str, parameters: dict | None = None) -> list[dict]:
    """Run a parameterized Cypher query; never interpolate user input into the
    query text -- same discipline as parameterized SQL."""
    with driver.session() as session:
        return [record.data() for record in session.run(query, parameters or {})]


# Constraint = uniqueness + backing index; MERGE matches-or-creates, so
# ingestion scripts are safe to re-run (unlike CREATE, which always inserts).
run_query("CREATE CONSTRAINT person_id IF NOT EXISTS FOR (p:Person) REQUIRE p.id IS UNIQUE")
run_query("CREATE CONSTRAINT company_id IF NOT EXISTS FOR (c:Company) REQUIRE c.id IS UNIQUE")

run_query(
    """
    MERGE (a:Person {id: $pid}) SET a.name = $name
    MERGE (c:Company {id: $cid}) SET c.name = $cname
    MERGE (a)-[r:WORKS_AT]->(c) SET r.role = $role
    """,
    {"pid": "p1", "name": "Alice", "cid": "c1", "cname": "Acme Robotics", "role": "Engineer"},
)

# Multi-hop MATCH: companies reachable within 1..3 hops of KNOWS/WORKS_AT edges.
# max_hops here is a fixed internal int, not raw user input, so interpolating
# it into the pattern (relationship type/bounds can't be query parameters) is safe.
rows = run_query(
    """
    MATCH (start:Person {name: $name})-[:KNOWS|WORKS_AT*1..3]->(reached)
    WHERE reached:Company
    RETURN DISTINCT reached.name AS company
    """,
    {"name": "Alice"},
)

GraphRAG

GraphRAG has two halves: an LLM extracts (subject, relation, object) triples from unstructured text, which are persisted as graph edges; then, per question, a bounded-hop subgraph around the relevant entity is retrieved, flattened into short triple lines, and passed to the LLM as grounding context — traceable to specific facts, unlike a wall of retrieved prose. Relationship types extracted by the LLM must be validated (alphanumeric allow-list) before being interpolated into Cypher, since they can't be passed as parameters.

import anthropic
client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-5",
    max_tokens=1024,
    system=f"Answer using only these graph facts:\n{context}",  # context = flattened subgraph rows
    messages=[{"role": "user", "content": question}],
)
Trade-off

LangChain's GraphCypherQAChain lets the LLM write Cypher itself from the schema (flexible, needs allow_dangerous_requests=True). The hand-rolled pattern above always runs a fixed, reviewed, bounded-hop query — prefer it in production; reach for the chain during prototyping.

3. Production & Integration

The common integration shape is a FastAPI endpoint: entity in, traversed subgraph out as JSON.

Client ──HTTP GET /graph-search──▶ FastAPI ──Cypher traversal──▶ Neo4j ──▶ JSON

@app.get("/graph-search")
def graph_search(entity: str, hops: int = 2) -> dict:
    if not 1 <= hops <= 4:
        raise HTTPException(400, "hops must be between 1 and 4")
    query = f"""
        MATCH path = (start:Entity {{name: $name}})-[*1..{hops}]-(other:Entity)
        UNWIND relationships(path) AS rel
        RETURN DISTINCT startNode(rel).name AS from_entity, type(rel) AS relation,
               endNode(rel).name AS to_entity
    """
    rows = run_query(query, {"name": entity})
    if not rows:
        raise HTTPException(404, f"No graph data for '{entity}'")
    return {"entity": entity, "facts": rows}

For GraphRAG assistants, the strongest pattern is hybrid retrieval: a vector index finds topically relevant candidate entities, then graph traversal pulls the structured relationships around them — vector search answers "what's relevant," the graph answers "how is it connected."

  • Constraints/indexes — every property used in a MATCH filter needs a backing index; uniqueness constraints provide both integrity and index.
  • Query profiling — prefix with EXPLAIN to see the plan, PROFILE to run it and see actual row counts/db hits per operator; catch label scans that should be index seeks.
  • Clustering/Aura — a causal cluster (self-managed) or Neo4j Aura (managed) gives a leader for writes plus followers/read replicas for scaling and failover.
  • Graph modeling effort — deciding what's a node vs. a property vs. a relationship is a real, continuous design cost that dominates database tuning; budget time for it before writing ingestion code.
  • Caching — cache expensive, frequently-repeated traversal results at the application layer; Neo4j's page cache handles hot data, but repeated multi-hop queries still benefit from an external cache.

Graph-based retrieval answers "what's connected, and by what path?" rather than "what's similar?" — Neo4j and Cypher provide the storage and pattern-matching traversal, including bounded multi-hop walks in a single query. Chapter 6

Vectorless Retrieval

1. Definition

Vectorless retrieval skips embeddings and vector databases altogether. You hand an LLM a list of candidate documents — just their paths, titles, or short snippets — and let it reason about which ones probably answer the question. The documents it picks are then loaded in full and passed to an LLM again to write a grounded, cited answer. In other words, relevance comes from the model actually reading and understanding the text, not from cosine-similarity math. The trade-off: you avoid all the moving parts of an embedding pipeline (re-embedding every time a document changes, keeping a vector index in sync, tuning chunk sizes) but you pay more for LLM calls. That makes it a genuinely good default for small knowledge bases that change often — not just a toy technique.

2. Development

The canonical pipeline has three steps, all on the official anthropic SDK: (1) scan a docs/ folder into cheap candidate metadata (path + snippet, no LLM calls); (2) ask an LLM to select the relevant files, with structured output; (3) load only the selected files' full text and ask an LLM to answer with citations.

Note

Use claude-haiku-4-5-20251001 for the cheap selection step (runs on every query) and claude-sonnet-5 for the answer step (needs real reading comprehension). Do not substitute retired identifiers such as claude-3-5-sonnet. 

"""vectorless_search.py - scan docs/, LLM selects files, LLM answers with citations.

No embeddings, no vector DB -- relevance and reading are both done by an LLM.
    python vectorless_search.py "How do I configure retries?"
"""
from __future__ import annotations

import sys
from pathlib import Path

from anthropic import Anthropic
from pydantic import BaseModel, Field

SELECT_MODEL = "claude-haiku-4-5-20251001"   # cheap filtering -- runs on every query
ANSWER_MODEL = "claude-sonnet-5"              # reading + reasoning -- stronger model
SNIPPET_CHARS = 400


class RankedFile(BaseModel):
    path: str
    relevance: float = Field(ge=0.0, le=1.0)
    reason: str


class RetrievalResult(BaseModel):
    selected: list[RankedFile]


def scan_folder(root: str | Path) -> list[tuple[str, str]]:
    """Cheap metadata-only scan: (relative path, snippet). No LLM calls, no vectors."""
    root = Path(root)
    return [
        (str(p.relative_to(root)), p.read_text(encoding="utf-8", errors="ignore")[:SNIPPET_CHARS])
        for p in sorted(root.rglob("*.md"))
    ]


def select_relevant_files(
    client: Anthropic, query: str, candidates: list[tuple[str, str]], top_k: int = 5
) -> list[RankedFile]:
    """LLM picks the most relevant files from cheap metadata -- structured output."""
    listing = "\n".join(f"PATH: {p}\nSNIPPET: {s}\n---" for p, s in candidates)
    response = client.messages.parse(
        model=SELECT_MODEL,
        max_tokens=1024,
        messages=[{
            "role": "user",
            "content": f"Select up to {top_k} files relevant to: {query}\n\n{listing}",
        }],
        output_format=RetrievalResult,
    )
    ranked = sorted(response.parsed_output.selected, key=lambda r: r.relevance, reverse=True)
    return ranked[:top_k]


def answer_query(client: Anthropic, root: str | Path, query: str, paths: list[str]) -> str:
    """Load only the selected files' full text and ask for a grounded, cited answer."""
    root = Path(root)
    context = "\n\n".join(
        f"=== FILE: {p} ===\n{(root / p).read_text(encoding='utf-8', errors='ignore')}" for p in paths
    )
    msg = client.messages.create(
        model=ANSWER_MODEL,
        max_tokens=2048,
        system="Answer using ONLY the provided documents. Cite every claim as [path].",
        messages=[{"role": "user", "content": f"DOCUMENTS:\n{context}\n\nQUESTION: {query}"}],
    )
    return next(b.text for b in msg.content if b.type == "text")


if __name__ == "__main__":
    client = Anthropic()  # reads ANTHROPIC_API_KEY
    query = " ".join(sys.argv[1:]) or "How do I get started?"

    candidates = scan_folder("docs")
    ranked = select_relevant_files(client, query, candidates)
    if not ranked:
        print("No relevant documents found.")
    else:
        for r in ranked:
            print(f"  selected: {r.path}  (relevance={r.relevance:.2f}) -- {r.reason}")
        print("\n--- ANSWER ---\n")
        print(answer_query(client, "docs", query, [r.path for r in ranked]))
Note

client.messages.parse(..., output_format=RetrievalResult) validates the model's JSON against the Pydantic schema automatically — the recommended way to get structured output from the Claude API, with no manual json.loads step and no risk of stray prose around the JSON.

Hierarchical retrieval scales this to larger corpora: a first pass asks the LLM to pick relevant top-level folders from a short list, then the file-level selector above runs only within those folders — no single prompt ever sees the whole corpus, however large it grows. Agentic retrieval goes further, giving the model tools (list_files, read_file, grep) and looping via the standard Anthropic tool-use pattern until it produces a final answer — useful when relevance depends on content a metadata snippet can't reveal, such as "which file mentions the v2 API?"

3. Production & Integration

Wrapping the pipeline in a FastAPI endpoint turns it into a service any client can call. This single /ask endpoint is both retriever and generator — no separate retrieval service or vector-store client to stand up, which makes it a good fit for small, self-contained knowledge bases. 

"""service.py - vectorless retrieval as a FastAPI endpoint."""
from fastapi import FastAPI
from pydantic import BaseModel
from anthropic import Anthropic

from vectorless_search import scan_folder, select_relevant_files, answer_query

app = FastAPI(title="Vectorless Retrieval Service")
client = Anthropic()
DOCS_ROOT = "docs"


class AskRequest(BaseModel):
    query: str
    top_k: int = 5


@app.post("/ask")
def ask(req: AskRequest) -> dict:
    candidates = scan_folder(DOCS_ROOT)
    ranked = select_relevant_files(client, req.query, candidates, req.top_k)
    if not ranked:
        return {"answer": "No relevant documents found.", "sources": []}
    sources = [r.path for r in ranked]
    return {"answer": answer_query(client, DOCS_ROOT, req.query, sources), "sources": sources}

Query --> LLM Retriever (haiku, cheap) --> LLM Answerer (sonnet, cited) --> Response
  • Context-window limits. The candidate listing (titles + snippets) must fit comfortably in context; past a few hundred documents, switch to hierarchical retrieval so no single prompt sees the whole corpus.
  • Prompt caching. Put the rendered candidate listing behind a cache_control breakpoint so repeated queries against an unchanged docs/ folder reuse the cached prefix instead of re-billing full price.
  • Cost and latency. Every query pays for candidate listing, selection reasoning, and full-document reading — log token usage per query and cap max_tokens and candidate count deliberately.
  • Guardrails. Require citations in every answer, reject answers citing a path outside the selected set, and confine any agentic file tools to the docs root.
  • When to graduate to embeddings. Once the corpus reaches the low thousands of documents, or the candidate listing routinely eats a meaningful fraction of the context window, switch to embedding-based retrieval instead of scaling hierarchical passes indefinitely.

5. Summary

It is the right default for small, frequently-changing knowledge bases, but the wrong choice once a corpus grows into the thousands of documents or a product needs sub-second, high-throughput answers, at which point embedding-based retrieval takes over — often alongside it, for the large, stable bulk of a corpus.

Chapter 7

Federated Retrieval

1. Definition

Federated retrieval is a coordination layer that sends one query out to many independent sources at the same time — GitHub, Slack, Notion, a vector store, an internal database — without first copying all their data into one shared index. Each source replies in its own format; the federated layer then reshapes every response into one common schema and merges the separate rankings into a single ordered list. The key idea is that each source keeps its own storage, its own search method, and its own notion of relevance — the federated layer owns only the query (fan it out) and the merge (combine the results), never the data itself. Reach for it when the sources genuinely can't or shouldn't be combined into one place — different owners, different access rules, or different update schedules. If everything already lives in one system, a single good index over that system is simpler.

2. Development

A federated search library needs four pieces: a shared Result/Source contract, one or more connectors that implement it, a concurrent federator with per-source timeouts, and Reciprocal Rank Fusion to merge the results.

Schema and Source Protocol

Every connector normalizes into Result and implements Source. The federator only ever talks to this interface — it never imports a connector's HTTP client or knows a source's native response shape.

from __future__ import annotations

from dataclasses import dataclass, field
from typing import Protocol, runtime_checkable


@dataclass(slots=True)
class Result:
    """A single normalized search result, regardless of origin system."""

    source: str             # e.g. "github", "notion", "elasticsearch"
    id: str                 # source-native identifier (unique within that source)
    title: str
    url: str
    snippet: str
    score: float = 0.0      # native relevance score -- NOT comparable across sources
    rank: int = 0            # 1-indexed rank within this source's own result list
    metadata: dict = field(default_factory=dict)


@runtime_checkable
class Source(Protocol):
    """Every connector implements this -- the only contract the federator relies on."""

    name: str

    async def search(self, query: str, limit: int = 10) -> list[Result]:
        """Return up to `limit` results, ranked best-first.

        Must not raise on an empty result set -- only on an unrecoverable
        connector failure (auth error, network failure). The federator
        turns that into a logged, non-fatal empty list.
        """
        ...

Connectors

GitHub — code search via the REST /search/code endpoint, authenticated with a personal access token:

from __future__ import annotations

import os

import httpx

from ..base import Result


class GitHubSource:
    """Searches GitHub code via the REST /search/code endpoint."""

    name = "github"

    def __init__(self, token: str | None = None, timeout: float = 5.0) -> None:
        self._token = token or os.environ["GITHUB_TOKEN"]
        self._timeout = timeout

    async def search(self, query: str, limit: int = 10) -> list[Result]:
        headers = {
            "Authorization": f"Bearer {self._token}",
            "Accept": "application/vnd.github+json",
        }
        params = {"q": query, "per_page": min(limit, 30)}

        async with httpx.AsyncClient(timeout=self._timeout) as client:
            resp = await client.get(
                "https://api.github.com/search/code",
                headers=headers,
                params=params,
            )
            resp.raise_for_status()
            payload = resp.json()

        results: list[Result] = []
        for rank, item in enumerate(payload.get("items", [])[:limit], start=1):
            results.append(
                Result(
                    source=self.name,
                    id=str(item["sha"]),
                    title=item["path"],
                    url=item["html_url"],
                    snippet=item["repository"]["full_name"],
                    score=item.get("score", 0.0),
                    rank=rank,
                )
            )
        return results

Federator: Concurrent Fan-Out With Timeouts

The federator runs every source's search() concurrently, bounds each with a timeout, and turns any failure into an empty result for that source rather than an exception that kills the whole request:

from __future__ import annotations

import asyncio
import logging

from .base import Result, Source

logger = logging.getLogger("federated_search")


async def _search_one(source: Source, query: str, limit: int, timeout: float) -> list[Result]:
    """Run one source under a hard timeout; never raises."""
    try:
        return await asyncio.wait_for(source.search(query, limit), timeout=timeout)
    except TimeoutError:
        logger.warning("source %s timed out after %.1fs", source.name, timeout)
        return []
    except Exception as exc:  # noqa: BLE001 -- a connector failure must not propagate
        logger.warning("source %s failed: %s", source.name, exc)
        return []


async def federated_search(
    sources: list[Source],
    query: str,
    limit_per_source: int = 10,
    timeout: float = 3.0,
) -> dict[str, list[Result]]:
    """Fan out `query` to every source concurrently.

    A slow or failing source degrades results for itself only -- it
    never blocks or drops the others. Total latency tracks `timeout`,
    not the sum of every source's response time.
    """
    tasks = [_search_one(source, query, limit_per_source, timeout) for source in sources]
    per_source_results = await asyncio.gather(*tasks)
    return {
        source.name: results
        for source, results in zip(sources, per_source_results, strict=True)
    }
Warning

Never swallow a source failure without logging it — an unlogged empty list looks identical to "this source legitimately had no matches," and nobody notices a broken connector until a user asks why one source never shows up.

Fusion: Reciprocal Rank Fusion

Native scores aren't comparable across systems — a GitHub relevance score and a BM25 score live on different, unbounded scales. RRF sidesteps that entirely by only looking at each result's rank within its own source's list:

score(doc) = sum over every source that returned doc of  1 / (k + rank_in_that_source)

from __future__ import annotations

from .base import Result


def reciprocal_rank_fusion(
    per_source_results: dict[str, list[Result]],
    k: int = 60,
) -> list[Result]:
    """Merge ranked lists from heterogeneous sources into one ranking.

    Ignores native scores entirely; works only from each result's rank
    within its own source's list, so every source's contribution lands
    on the same bounded scale. `k` (default 60, the standard value from
    the original RRF paper) dampens any single top rank from dominating.
    """
    fused_scores: dict[tuple[str, str], float] = {}
    result_by_key: dict[tuple[str, str], Result] = {}

    for source_name, results in per_source_results.items():
        for result in results:
            key = (source_name, result.id)
            fused_scores[key] = fused_scores.get(key, 0.0) + 1.0 / (k + result.rank)
            result_by_key[key] = result

    ranked_keys = sorted(fused_scores, key=lambda key: fused_scores[key], reverse=True)
    fused: list[Result] = []
    for key in ranked_keys:
        result = result_by_key[key]
        result.metadata["rrf_score"] = round(fused_scores[key], 5)
        fused.append(result)
    return fused

An optional rerank pass over the fused top-N can reorder by semantic judgment RRF can't express, using the Anthropic API (claude-sonnet-5) with a JSON-schema-constrained response for a guaranteed-parseable ordering. Frameworks ship equivalent fan-out-and-fuse blocks too: LangChain's EnsembleRetriever combines multiple retrievers with RRF-style fusion, and LlamaIndex's QueryFusionRetriever dispatches across multiple indexes and fuses the results.

3. Production & Integration

The most common shape is a single async endpoint fronting an enterprise assistant that searches every internal and third-party system — Slack, Notion, GitHub, Confluence — through one interface, regardless of which tool holds the answer:

Client -> FastAPI /federated-search -> asyncio.gather(sources) -> RRF fusion -> JSON response

    fused = reciprocal_rank_fusion(per_source)[:top_n]
    return {
        "query": q,
        "results": [
            {
                "source": r.source,
                "title": r.title,
                "url": r.url,
                "score": r.metadata.get("rrf_score", r.score),
            }
            for r in fused
        ],
    }
  • Per-source auth & secrets. Each connector's credential is scoped to that source only and pulled from a secrets manager; rotate them independently.
  • Permission-aware filtering. Pass the requesting user's own credentials (or an on-behalf-of token) into each connector so the source itself enforces access control — the federator must never widen access.
  • Latency bounded by the slowest source. Per-source timeout plus asyncio.gather keep one degraded connector from stalling the request; return partial results instead of failing the whole call.
  • Caching. Cache normalized per-source results (not just the fused response), keyed by (source, query, permission-scope), with TTLs short enough for fast-moving sources.
  • Connector maintenance. Each third-party API can change its schema or auth model on its own timeline — give every connector its own health check and owner.
Warning

Permission-aware filtering is the easiest thing to get wrong: caching across users, or querying a source with a privileged service account instead of the requester's own scope, silently turns the federator into a way to leak data a user could never reach directly.
