Chapter 1

Traditional (Lexical) Search

1. Definition

Lexical search (also called keyword or sparse search) is the classic "match the words" approach — the same idea behind Ctrl+F or a plain search bar. It looks for the exact words from your query inside the documents. To do that quickly, it builds an inverted index: picture the index at the back of a textbook that maps each word to the list of pages where it appears. It then sorts the matching documents with BM25, a scoring formula that (a) rewards documents that use your search words more often (term frequency), (b) gives extra weight to rare, specific words (inverse document frequency), and (c) stops very long documents from winning just because they contain more words (length normalization). Because it only matches the literal text and ignores meaning, it's the right tool when things must match exactly — product SKUs, error codes, function names, and IDs — where "close enough" is simply wrong.

Note

"Sparse" and "lexical" mean almost the same thing here. Sparse describes how the text is stored — as a long list of word counts that is mostly zeros, since most words never appear in any given document. Lexical describes the strategy — matching the exact words. BM25 is the standard formula for scoring those matches.

2. Key Features

  • Exact token matching — no hallucinated relevance; a term either appears in a document or it doesn't.
  • Inverted index — O(1) amortized term lookup via a hash map, no full corpus scan.
  • BM25/TF-IDF ranking — tunable, well-understood, no training step.
  • Very low latency — single-digit ms queries at scale, no GPU in the critical path.
  • Cheap infrastructure — runs on commodity CPU/RAM only.
  • Interpretable — every score is explainable from term frequency, IDF, and length norm.
  • Battle-tested at scale — decades of Lucene-derived production hardening (Elasticsearch, OpenSearch, Solr).

3. How It Works

Lexical search has two simple phases. At index time, it reads every document and builds an inverted index — a lookup table from each word to the list of documents that contain it. At query time, it splits your query into words, looks each one up in that table, and collects the documents that match. Here is the whole idea in a few lines: 

# The essence: an inverted index maps each word -> the docs that contain it.
docs = {
    "d1": "connection reset error",
    "d2": "database connection pool",
    "d3": "user login timeout",
}

index = {}                                  # word -> set of doc ids
for doc_id, text in docs.items():
    for word in text.split():
        index.setdefault(word, set()).add(doc_id)

# A query just looks up its words and gathers the docs that mention them.
query = "connection error"
hits = set()
for word in query.split():
    hits |= index.get(word, set())

print(hits)                                 # {'d1', 'd2'}  <- both mention "connection"

Matching tells you which documents contain the words; BM25 decides the order. You don't need the formula to get the intuition — it rewards three things: documents that use your search words more often, words that are rare and specific (matching "ERR_CONN_RESET" says more than matching "the"), and it stops very long documents from ranking high just because they contain more words overall.

In one sentence

Build a word → documents lookup table once, then answer every query by looking up its words and ranking the matches with BM25 — no model, no training, no GPU.

4. When to Use It

Lexical search is the right default whenever the exact words matter more than their meaning.

Good fit — reach for it when…

  • Exact identifiers must match. Product SKUs, error codes, order numbers, function/API names, legal citations — anything where "close enough" is wrong.
  • Users type the words they expect to find. Code search, log search, documentation search, e-commerce keyword filters.
  • You need it fast, cheap, and explainable. Single-digit-millisecond queries on ordinary CPUs, and every score can be explained.
  • You want a strong, simple baseline. It's the retriever to beat, and often the first stage that a smarter method reranks.

Poor fit — look elsewhere when…

  • Users phrase things differently than your documents. A search for "car" won't find a page that only says "automobile" → use dense vector search.
  • Meaning and paraphrase matter. FAQ-style or conversational questions → dense or learned sparse retrieval.
  • You need cross-language search. Query in one language, documents in another → multilingual embeddings.
Rule of thumb

If a user would type the exact word they expect to see in the result, lexical search wins. When they'd describe it in their own words instead, reach for dense search — and in practice, running both together ("hybrid") is a very strong default.

It's the right default for exact identifier lookup, code search, and any workload where literal precision matters, but it cannot bridge synonyms or paraphrase — the gap dense vector search closes in the next chapter.

Chapter 2

Dense Vector Search

1. Definition

Dense vector search turns each document and each query into an embedding — a fixed-length list of numbers (a "vector") produced by a neural network. The easiest way to picture it: each vector is a point in space, and things with similar meaning sit close together. Finding relevant text then becomes a geometry problem — find the document points nearest to the query point, measured with cosine similarity or dot product. The big advantage is that it captures meaning, not just words, so "car" and "automobile" land near each other even though they share no letters. That's exactly what powers semantic search and Retrieval-Augmented Generation (RAG). Checking the query against every stored vector one by one is accurate but slow once you have millions or billions of them, so real systems use Approximate Nearest Neighbor (ANN) indexes (HNSW, IVF, PQ) that find the near-matches far faster, accepting a tiny drop in accuracy for a big speed gain.

2. Key Features

  • Semantic matching — retrieves by meaning, handling synonyms, paraphrase, and cross-lingual queries.
  • Embedding models — pretrained bi-encoders (sentence-transformers, OpenAI, Cohere) map text to fixed-size vectors in one forward pass.
  • Similarity metrics — cosine, dot product, and Euclidean (L2) distance quantify vector closeness.
  • ANN indexes — HNSW, IVF, and Product Quantization enable sub-linear search at scale for a small recall cost.
  • Multilingual support — multilingual embedding models place equivalent sentences from different languages near each other.
  • RAG backbone — supplies the "R" in RAG, grounding LLM output in retrieved, citable text.
  • Millisecond retrieval at scale — ANN indexes return top-k from billions of vectors in single-digit milliseconds.

3. How It Works

Dense search also has two phases. At index time, an embedding model reads each document and turns it into a vector — a list of numbers that captures its meaning — and those vectors are stored. At query time, the same model turns your query into a vector, and the system returns the stored vectors that point in the most similar direction, measured with cosine similarity. The key insight is that similar meaning produces similar vectors, even when the words are completely different: 

import numpy as np

# Pretend an embedding model already turned each word into a vector.
# Similar meaning -> vectors that point in a similar direction.
vectors = {
    "car":        np.array([0.90, 0.10, 0.20]),
    "automobile": np.array([0.88, 0.12, 0.18]),   # <- almost identical to "car"
    "banana":     np.array([0.10, 0.90, 0.30]),
}

def cosine(a, b):
    return a @ b / (np.linalg.norm(a) * np.linalg.norm(b))

query = vectors["car"]
for word, v in vectors.items():
    print(word, round(cosine(query, v), 3))
# car 1.0 | automobile 0.999 | banana 0.421   <- meaning, not spelling

Real embedding models use hundreds or thousands of dimensions instead of three, but the principle is identical. Comparing your query against every stored vector one by one is accurate but slow once you have millions of them, so production systems use an ANN index (HNSW, IVF) that finds the closest vectors almost instantly, trading a tiny bit of accuracy for a huge speed gain.

In one sentence

Turn text into vectors whose direction encodes meaning, then answer a query by finding the nearest vectors — this is what lets "car" match "automobile," and it's the engine behind semantic search and RAG.

4. When to Use It

Dense search shines whenever meaning matters more than the exact words a user typed.

Good fit — reach for it when…

  • Users ask in their own words. Help centers, FAQ search, support bots — the query rarely uses the exact wording of the answer.
  • Synonyms and paraphrase are common. "How do I cancel my plan?" should find a doc titled "Ending a subscription."
  • You're building RAG. Dense retrieval is the standard way to fetch the passages that ground an LLM's answer.
  • You need "find similar" or multilingual search. Related articles, similar products, or matching a query in one language to documents in another.

Poor fit — look elsewhere when…

  • Exact identifiers must match. SKUs, error codes, and function names are better (and cheaper) served by lexical search.
  • The corpus is tiny or changes constantly. Re-embedding on every edit may not be worth it → consider vectorless retrieval.
  • You need fully explainable scores or have no infra for a model. A cosine number is harder to justify than a BM25 term match.
Rule of thumb

If users describe what they want in their own words, go dense. It pairs extremely well with lexical search — a "hybrid" of the two covers both meaning and exact matches, and is the safest default for most search products.

Chapter 3

Learned Sparse Retrieval

1. Definition

Learned sparse retrieval is a hybrid: it keeps the fast, word-based machinery of keyword search but lets a neural network decide how important each word is, instead of a hand-written formula like TF-IDF or BM25. The best-known model is SPLADE (SParse Lexical AnD Expansion). For every document and query it produces a sparse {term_id: weight} list — a weight for just a handful of words out of the full ~30,000-word vocabulary, and zero for all the rest. Its clever trick is term expansion: it can switch on related words that never actually appear in the text (for example, "heart attack" also lights up "cardiac" and "myocardial"), giving you some of the meaning-awareness of dense search. And because only a few hundred words have a nonzero weight, the output drops straight into the same proven keyword-search engines (Lucene, Elasticsearch, OpenSearch, Qdrant sparse vectors) — no special vector (ANN) index required.

Note

Think of learned sparse retrieval as the middle ground between two extremes: classic keyword search (BM25 — fast and exact, but blind to meaning) and dense vector search (smart about meaning, but a "black box" that needs a special vector index). That's why it's nicknamed "neural lexical search" — a neural network judges relevance, but the result stays a plain, human-readable list of weighted words.

2. Key Features

  • Term expansion — activates semantically related vocabulary terms absent from the source text, closing BM25's vocabulary-mismatch gap.
  • Sparse weighted vectors — typically 100–400 nonzero entries out of a ~30K vocabulary.
  • Inverted-index compatible — nonzero weights map onto postings lists; scoring is a dot product over shared terms, no HNSW graph or vector DB required.
  • Interpretable — decode token ids back to strings to see exactly which terms fired and why, useful for debugging relevance.
  • Strong first-stage recall — matches or beats BM25 and competes with dense bi-encoders on BEIR benchmarks.
  • Drop-in for existing search infra — bolts onto Lucene/Elasticsearch/OpenSearch via rank_features or native neural-sparse fields.

3. How It Works

Learned sparse retrieval looks like keyword search, but a neural network does two extra jobs. First, it decides how important each word is (instead of a fixed formula). Second, it can add related words that never appeared in the text — this is term expansion, and it's what closes keyword search's biggest weakness. The output is just a list of {word: weight} pairs, so scoring is the same dot product a keyword engine already does: 

# A learned-sparse model turns text into {word: importance}, and can switch on
# related words that never appeared in the text ("expansion").
doc = {                       # a document about a heart attack...
    "heart": 1.8, "attack": 1.6,
    "cardiac": 1.1, "myocardial": 0.9,   # <- EXPANDED: never written, still added
}
query = {"heart": 1.7, "attack": 1.5}

# Score = add up the weights for words the query and doc share (a dot product).
score = sum(w * doc.get(term, 0) for term, w in query.items())
print(round(score, 2))        # 5.46  -> strong match, and it would also match a
                              # doc that only said "cardiac" thanks to expansion

Because the result is still a bag of weighted words, it drops straight into the same inverted-index engines (Elasticsearch, OpenSearch, Lucene, Qdrant) that power BM25 — no separate vector database required. And unlike a dense vector, you can read the weights directly to see exactly which words fired and why.

In one sentence

It's keyword search with a brain: a neural net picks the important words and adds related ones, but the output stays a plain, human-readable list of weighted terms your existing search engine already understands.

4. When to Use It

This is the method for teams that already run a keyword-search engine and want more relevance without replacing it.

Good fit — reach for it when…

  • You already run Elasticsearch / OpenSearch / Lucene. You get semantic-style recall by adding a field, not by standing up a whole new vector database.
  • You need interpretable, debuggable relevance. You can print the exact terms that fired — valuable in regulated or high-stakes domains.
  • Vocabulary mismatch hurts you. Medical, legal, or technical corpora where users and documents use different words for the same thing.
  • You want a strong first stage. Often fused with BM25 and/or dense retrieval for the best of all three.

Poor fit — look elsewhere when…

  • You have no existing search infrastructure. For a small project, a plain dense or vectorless setup is less to build.
  • Latency budgets are extreme. Encoding the query adds a neural-network forward pass that raw BM25 doesn't.
  • You need full paraphrase-level meaning. When matches depend on deep semantics beyond word overlap → dense or multi-vector.
Rule of thumb

Choose learned sparse when you have mature keyword-search infrastructure and want to make it smarter without ripping it out — you keep the speed, tooling, and interpretability of lexical search and gain semantic recall.

6. Summary

Learned sparse retrieval, exemplified by SPLADE, replaces BM25's fixed term-frequency heuristics with a neural model that expands documents/queries into semantically related vocabulary while keeping output sparse and inverted-index-compatible. Chapter 4

Multi-Vector Search

1. Definition

Most vector search squeezes a whole document down into a single embedding. Multi-vector search — made popular by ColBERT and extended by BGE-M3 — instead keeps one embedding per token (roughly, per word), so a document becomes a whole set of vectors rather than one averaged-out vector. It compares them using late interaction: the query and the document are each encoded on their own first (which is fast and can be cached), and only afterward are their token vectors matched up. The scoring rule is MaxSim: for each query word, find its best-matching word in the document, then add up those best matches — score(Q, D) = Σ_i max_j cos(q_i, d_j). Keeping every token means the fine details aren't blurred away by averaging, which is why it's the top choice for long, many-topic documents and the go-to second-pass reranker in high-precision RAG pipelines.

2. Key Features

  • Token-level embeddings — every token gets its own vector instead of being pooled away.
  • Late interaction / MaxSim — relevance is a sum of per-token maximum similarities, not a single dot product.
  • High retrieval accuracy — consistently outperforms single-vector dense retrieval on precision-sensitive benchmarks (BEIR, LoTTE).
  • Strong on long, multi-topic documents — each token contributes independently instead of being averaged away.
  • Excellent reranking signal — the standard second stage after a cheap first-stage retriever (BM25 or single-vector dense).
  • Higher storage cost — one vector per token instead of one per document multiplies index size roughly by token count.

3. How It Works

Ordinary dense search squeezes a whole document into a single vector, which averages all its details together. Multi-vector search keeps one vector per word instead, so nothing is blurred away. To compare a query and a document it uses MaxSim: for each query word, find its best-matching word anywhere in the document, then add up those best matches. That's the entire scoring rule: 

import numpy as np

# One vector PER WORD (not one per document). Tiny 2-D vectors for illustration.
query = np.array([[1.0, 0.0],           # query word 1
                  [0.0, 1.0]])          # query word 2
doc   = np.array([[0.9, 0.1],           # doc word 1  (matches query word 1)
                  [0.1, 1.0],           # doc word 2  (matches query word 2)
                  [0.5, 0.5]])          # doc word 3

sims = query @ doc.T                    # similarity of every query word to every doc word
best_per_query_word = sims.max(axis=1)  # each query word keeps only its best match
print(best_per_query_word.sum())        # 1.9  <- the MaxSim score

Keeping every word's vector is what makes it so precise on long, many-topic documents — but it also means far more vectors to store. So in practice it is used as a reranker: a cheap first-stage retriever (keyword or dense) fetches ~100 candidates, and MaxSim carefully re-scores just those to pick the true top few.

In one sentence

Compare documents word-by-word instead of as a single averaged blob — far more precise, but heavy enough that it's usually a second-pass reranker rather than your only retriever.

4. When to Use It

Multi-vector search is a precision tool — reach for it when "almost right" isn't good enough.

Good fit — reach for it when…

  • Precision is critical. The answer must come from the exactly right passage, not merely a topically-close one.
  • Documents are long and cover many topics. One averaged vector smears them together; per-word vectors keep each point distinct.
  • You're reranking in a RAG pipeline. Use it as the second stage to pick the chunk that truly contains the answer before it reaches the LLM.
  • You have GPU budget and room for a bigger index. The quality gain is real when you can pay for it.

Poor fit — look elsewhere when…

  • Storage, latency, or cost are tight. One vector per word is roughly 10–100× larger than a single-vector dense index.
  • A simpler retriever already meets your quality bar. If dense or lexical is "good enough," the extra complexity isn't worth it.
  • The corpus is small. With few, short documents the precision gain is rarely noticeable.
Rule of thumb

Treat it as a precision booster, not your only retriever: let a cheap first stage narrow the field, then use multi-vector MaxSim to rerank a small candidate set for the final, high-quality answer.

Chapter 5

Graph-Based Retrieval

1. Definition

Graph-based retrieval stores your data as a property graph — nodes for things (people, companies, products) joined by labeled, directional edges that describe how they relate (WORKS_AT, PRODUCES), where both nodes and edges can carry extra key-value details. To answer a question, it follows the connections outward from a starting node instead of measuring how similar two pieces of text are. That makes it perfect for "who is connected to what" questions — "who else works where Alice used to work?", "is this transaction linked to a known fraud ring?" — which are answered by hopping across relationships. A vector index simply can't do this: it can tell you two documents are on similar topics, but not that two specific entities are exactly two hops apart. GraphRAG brings this to LLM apps: instead of dumping the top-k similar text chunks into the prompt, it pulls out a small, structured, explainable slice of the graph and hands that to the model as context.

2. Key Features

  • Nodes, edges, properties — labeled entities and typed, directed relationships, both with arbitrary key-value attributes.
  • Relationship traversal — retrieval is a graph walk (Cypher MATCH), not nearest-neighbor search.
  • Multi-hop reasoning — one query follows chains of relationships several hops deep in a single round trip.
  • Explainable paths — every result comes with the exact node/edge path that produced it, not a black-box similarity score.
  • Cypher query language — declarative pattern matching, roughly SQL for graphs.
  • Strength on connected data — fraud rings, recommendation networks, supply chains, and knowledge graphs are naturally graph-shaped.

3. How It Works

Graph retrieval stores your data as nodes (things) connected by relationships (how they relate). To answer a question you don't measure similarity — you follow the connections from a starting point. Here is a "who else works where Alice works?" question answered by hopping one step out and one step back: 

# Data as connections: (thing) -[relationship]-> (thing)
edges = [
    ("Alice", "WORKS_AT", "Acme"),
    ("Bob",   "WORKS_AT", "Acme"),
    ("Carol", "WORKS_AT", "Globex"),
]

# Hop 1: where does Alice work?  Hop 2: who else works there?
alice_orgs = [org for a, rel, org in edges if a == "Alice" and rel == "WORKS_AT"]
coworkers  = [p for p, rel, org in edges
              if rel == "WORKS_AT" and org in alice_orgs and p != "Alice"]

print(coworkers)          # ['Bob']  <- found by relationship, not by text similarity

Real systems use a graph database like Neo4j and its query language, Cypher, which expresses that same multi-hop walk in a single readable line: 

MATCH (alice:Person {name: 'Alice'})-[:WORKS_AT]->(org)<-[:WORKS_AT]-(peer)
RETURN peer.name

GraphRAG brings this to LLM apps: instead of dumping the top-k similar text chunks into a prompt, it retrieves a small, connected slice of the graph and hands the model those exact facts — traceable and explainable.

In one sentence

Store how things are connected, then answer questions by walking the connections — the only method that can say two entities are exactly N relationships apart, not just "topically similar."

4. When to Use It

Use a graph when the value lives in the connections between items, not in the items themselves.

Good fit — reach for it when…

  • The question is about relationships. "Who is linked to whom?", "what's two hops away?", "what path connects these two?"
  • Your domain is naturally a network. Fraud rings, recommendation graphs, supply chains, org charts, knowledge graphs.
  • You need multi-hop answers. "Is this transaction linked to a known-bad account through any chain of transfers?" — impossible for a similarity search.
  • Explainability matters. Every answer comes with the exact path of nodes and edges that produced it.

Poor fit — look elsewhere when…

  • Your data is independent documents. A pile of articles with no meaningful links → dense or lexical search is far simpler.
  • You just need "find similar text." Similarity is exactly what vector search does; a graph adds cost for no gain.
  • You can't invest in modeling. Deciding what's a node vs. a property vs. a relationship is ongoing design work — budget for it.
Rule of thumb

If your hardest questions start with "who/what is connected to…" or "how are these two related," use a graph. For LLM apps, the strongest setup is hybrid: vector search finds the relevant entities, then the graph explains how they connect.

6. Summary

Graph-based retrieval answers "what's connected, and by what path?" rather than "what's similar?" — Neo4j and Cypher provide the storage and pattern-matching traversal, including bounded multi-hop walks in a single query. GraphRAG extends this to LLM apps by extracting triples, persisting them as a graph, and retrieving a small explainable subgraph as grounding context, at the ongoing cost of real graph-modeling effort — a cost worth paying when the domain (fraud, recommendations, supply chains, knowledge graphs) is inherently relational rather than a set of independent documents.

Chapter 6

Vectorless Retrieval

1. Definition

Vectorless retrieval skips embeddings and vector databases altogether. You hand an LLM a list of candidate documents — just their paths, titles, or short snippets — and let it reason about which ones probably answer the question. The documents it picks are then loaded in full and passed to an LLM again to write a grounded, cited answer. In other words, relevance comes from the model actually reading and understanding the text, not from cosine-similarity math. The trade-off: you avoid all the moving parts of an embedding pipeline (re-embedding every time a document changes, keeping a vector index in sync, tuning chunk sizes) but you pay more for LLM calls. That makes it a genuinely good default for small knowledge bases that change often — not just a toy technique.

2. Key Features

  • No embedding pipeline or vector database — nothing to provision, version, or re-run when documents change.
  • Always up to date — reads current files at query time; no stale index to refresh.
  • Leverages full LLM reasoning — relevance judgments benefit from language understanding, not dense-vector similarity.
  • Simple architecture — two or three Python files implement a working system, no extra services to deploy.
  • Higher inference cost per query — every query spends tokens on candidate listing, selection, and full-document reading.
  • Best for small, volatile corpora — personal knowledge bases, internal wikis, Obsidian-style vaults.

3. How It Works

Vectorless retrieval replaces the whole embed-store-search machine with two LLM steps. A cheap model selects which files are worth reading (from just their paths and a short snippet), then a stronger model reads those files in full and answers with citations. Relevance comes from the model actually understanding the text — no vectors anywhere: 

# No embeddings, no vector database. The LLM decides what's relevant, then reads it.
files = scan_folder("docs/")                 # [(path, first 400 chars), ...]  -- cheap, no model

picked = cheap_llm(                          # Step 1: pick the relevant files
    f"Which of these files answer the question: {question}?\n{files}"
)

answer = strong_llm(                         # Step 2: read them in full and cite
    f"Answer using ONLY these documents:\n{load(picked)}\n\nQuestion: {question}"
)

Because it reads the live files at query time, there is never a stale index to refresh — edit a document and the next question already sees the change. The cost is paid in tokens on every query instead of in infrastructure. For bigger folders, a hierarchical version first picks relevant sub-folders, then files, so no single prompt ever has to look at the entire corpus.

In one sentence

Let the LLM skim a list of files, choose the useful ones, and read them — trading indexing infrastructure for a bit more thinking per query, and staying perfectly up to date by construction.

4. When to Use It

Vectorless retrieval is the low-setup choice for small collections that change all the time.

Good fit — reach for it when…

  • The corpus is small. A personal notes vault, a team wiki, a repository's docs/ folder — up to a few hundred documents.
  • It changes constantly. No index to rebuild means edits are reflected instantly, with zero maintenance.
  • You want zero infrastructure. No embedding model, no vector database, no re-indexing job — a couple of files and an API key.
  • You're prototyping. A perfect MVP to prove value before investing in a full retrieval stack.

Poor fit — look elsewhere when…

  • The corpus is large. Thousands of documents won't fit in a prompt to list → switch to dense (embedding) retrieval.
  • You have high traffic or tight latency/cost budgets. Every query spends tokens reading documents, which adds up fast.
  • The content is large and stable. A one-time index amortizes far better than re-reading on every question.
Rule of thumb

Great for small and fast-changing. As the collection grows into the thousands of documents — or queries get frequent enough that token cost bites — graduate to embedding-based retrieval, often keeping vectorless for the volatile edges.

6. Summary

Vectorless retrieval replaces the embed-store-search pipeline with a read-reason-select loop: a cheap model selects candidate documents from metadata, and a stronger model reads them in full to produce a grounded, cited answer — no embedding model, no vector database, no re-indexing job. Chapter 7

Federated Retrieval

1. Definition

Federated retrieval is a coordination layer that sends one query out to many independent sources at the same time — GitHub, Slack, Notion, a vector store, an internal database — without first copying all their data into one shared index. Each source replies in its own format; the federated layer then reshapes every response into one common schema and merges the separate rankings into a single ordered list. The key idea is that each source keeps its own storage, its own search method, and its own notion of relevance — the federated layer owns only the query (fan it out) and the merge (combine the results), never the data itself. Reach for it when the sources genuinely can't or shouldn't be combined into one place — different owners, different access rules, or different update schedules. If everything already lives in one system, a single good index over that system is simpler.

2. Key Features

  • Query distribution (fan-out) — the query is dispatched to every source concurrently, not sequentially, so total latency tracks the slowest source, not the sum of all of them.
  • Result normalization — each source's native response shape is mapped into one common schema before anything downstream touches it.
  • Unified ranking (RRF) — results from sources with incomparable, unbounded relevance scales are merged into one ranked list via Reciprocal Rank Fusion.
  • Permission-aware retrieval — the federator returns only what the requesting user is authorized to see in each source, never widening access beyond what the source itself grants.
  • Heterogeneous back-ends — full-text indexes, vector stores, relational databases, or third-party REST APIs sit behind one uniform contract.
  • No data migration — nothing is copied into a central store; each source is queried live, so freshness and ownership stay exactly where they already are.

3. How It Works

Federated retrieval isn't a new search algorithm — it's a coordination layer over search engines you already have. It does three things: fan out the query to every source at the same time, normalize each response into one common shape, and merge the separate ranked lists into one. The tricky part is the merge: a GitHub score and a database score aren't comparable, so Reciprocal Rank Fusion (RRF) ignores raw scores and looks only at each result's rank: 

# Each source returns its OWN ranked list. Their scores aren't comparable,
# so we merge by rank position instead, using Reciprocal Rank Fusion.
results = {
    "github": ["A", "B", "C"],     # A ranked #1 here
    "slack":  ["B", "D"],          # B ranked #1 here
}

k, fused = 60, {}
for source, ranked in results.items():
    for rank, doc in enumerate(ranked, start=1):
        fused[doc] = fused.get(doc, 0) + 1 / (k + rank)   # add 1 / (k + rank)

print(sorted(fused, key=fused.get, reverse=True))
# ['B', 'A', 'C', 'D']  <- B wins: it appeared near the top of TWO sources

The fan-out runs concurrently, so total time tracks the slowest source, not the sum of all of them — and if one source is down, the others still return. Each system keeps its own data, its own search method, and its own permissions; the federated layer owns only the query and the merge, never the data.

In one sentence

Ask many independent systems at once and blend their rankings into one list, without ever copying their data into a shared index — the answer can live anywhere, and the user only sees one search box.

4. When to Use It

Federated retrieval is for when your knowledge is scattered across systems that must stay where they are.

Good fit — reach for it when…

  • The data lives in many places. An enterprise "search everything" assistant across GitHub, Slack, Notion, Drive, and internal databases at once.
  • Sources can't or shouldn't be merged. Different owners, different update schedules, or compliance rules that forbid copying data into one store.
  • Permissions differ per source. Query each system with the user's own credentials so it enforces its own access control — nothing is over-shared.
  • A central index would go stale. Live querying keeps every result as fresh as the source itself.

Poor fit — look elsewhere when…

  • Everything already lives in one system. Just index that one system well — a single index is simpler and faster than N connectors.
  • You can consolidate cheaply. If the data can move into one store, that usually beats maintaining a connector per source.
  • You need deep cross-source relevance tuning. RRF merges by rank only; it's robust but coarse, and can't express fine semantic judgments across systems.
Rule of thumb

Federate when the data must stay where it is; consolidate when it can move. The real work isn't the fan-out — it's honoring each source's permissions and keeping every connector healthy over time.

6. Summary

Federated retrieval unifies search across systems that stay independent — no migration, no shared index, no single owner of the data — by fanning the query out concurrently, normalizing every source's response into one schema, and fusing per-source rankings with Reciprocal Rank Fusion.
